# 接口文档

本文档统一按“用途 + 参数表 + 关键响应字段 + JSON 示例”的格式描述 10 个接口当前实现口径。

## 接口 1：协议文档基础校验

- 路由：`POST /api/data/validate_protocol_files`
- Content-Type：`application/json` 或 `multipart/form-data`
- 用途：校验协议文件格式、大小和可读性，建议在正式上传拆分前调用。支持直接传本机文件路径，也兼容传统上传文件。大文件支持后台异步校验与 PDF 按页批次处理。

### 请求参数

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `file_path` | string | 否 | 单个本地文件路径，与 `file/files/file_paths` 至少提供一种。 |
| `file_paths` | string[] | 否 | 多个本地文件路径，与 `file/files/file_path` 至少提供一种。 |
| `file` | binary | 否 | 单文件上传，与 `file_path/file_paths/files` 至少提供一种。 |
| `files` | binary[] | 否 | 多文件上传，与 `file_path/file_paths/file` 至少提供一种。 |
| `max_size_mb` | integer | 否 | 单文件大小上限，默认 `50`。 |
| `async` | boolean | 否 | 是否强制走后台任务；大文件也可能由服务端自动切换到后台模式。 |
| `page_batch_size` | integer | 否 | PDF 按页批次处理时每批页数，默认 `100`。 |
| `stream` | boolean | 否 | 是否启用 SSE 流式输出。 |

### 同步响应关键字段

| 字段 | 说明 |
|---|---|
| `data.summary.total_files` | 总文件数 |
| `data.summary.passed_files` | 通过数量 |
| `data.summary.failed_files` | 失败数量 |
| `data.results[]` | 每个文件的校验结果 |

### 异步相关接口

| 路由 | 说明 |
|---|---|
| `GET /api/data/validate_protocol_files/status?job_id=...` | 查询后台校验任务状态 |
| `GET /api/data/validate_protocol_files/stream?job_id=...` | 订阅后台校验任务流式进度 |

### 异步状态关键字段

| 字段 | 说明 |
|---|---|
| `data.job_id` | 后台任务 ID |
| `data.status` | 任务状态，典型值 `queued`、`running`、`completed`、`failed` |
| `data.stage` | 当前阶段 |
| `data.message` | 当前阶段描述 |
| `data.metadata.page_batch_size` | 当前页批次大小 |
| `data.metadata.processed_pages` | 已处理页数 |
| `data.metadata.total_pages` | 总页数 |
| `data.result` | 任务完成后的最终 JSON 结果 |

### JSON 请求示例

```json
{
  "file_paths": [
    "/data/protocols/k1.6.pdf",
    "/data/protocols/k1.7.pdf"
  ],
  "max_size_mb": 50,
  "page_batch_size": 100,
  "async": false
}
```

### JSON 响应示例

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "summary": {
      "total_files": 2,
      "passed_files": 2,
      "failed_files": 0
    },
    "results": [
      {
        "file_name": "k1.6.pdf",
        "file_path": "/data/protocols/k1.6.pdf",
        "valid": true,
        "file_type": "pdf",
        "message": "文件可正常读取"
      }
    ]
  }
}
```

### 同路由 SSE 流式输出

除后台任务专用的 `/status`、`/stream` 路由外，根入口 `POST /api/data/validate_protocol_files` 也支持直接开启 SSE。

可任选一种方式开启：

- 请求体中传 `stream=true`
- Query 参数传 `?stream=true`
- Header 传 `X-Stream-Response: true`

流式响应：

- Content-Type：`text/event-stream`
- 事件类型：`open`、`progress`、`heartbeat`、`result`、`error`、`close`

说明：

- `progress` 事件固定返回 `stage=started`
- `result` 事件中的 `body` 与 `result` 字段保存与同步模式相同的完整 JSON 返回内容

### SSE 流式输出示例

```text
event: open
data: {"task":"validate_protocol_files","message":"流式连接已建立","timestamp":1777000000.101}

event: progress
data: {"task":"validate_protocol_files","stage":"started","message":"协议文件校验已开始，正在解析文件内容","timestamp":1777000000.102}

event: result
data: {"task":"validate_protocol_files","stage":"completed","timestamp":1777000001.215,"status_code":200,"body":{"code":200,"message":"success","data":{"summary":{"total_files":2,"passed_files":2,"failed_files":0}}},"result":{"code":200,"message":"success","data":{"summary":{"total_files":2,"passed_files":2,"failed_files":0}}}}

event: close
data: {"task":"validate_protocol_files","message":"流式输出结束","timestamp":1777000001.216}
```

## 接口 2：文档上传拆分

- 路由：`POST /api/data/upload_split`
- Content-Type：`application/json` 或 `multipart/form-data`
- 用途：上传协议文档并拆分为块，同时在上传阶段立即创建对应的 PageIndex 索引。当前 PageIndex 对超大文档会自动按分片建索引，后续检索只扫候选分片，不再整本文档全扫。

### 请求参数

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `project_id` | string | 是 | 项目标识。 |
| `dataset_id` | string | 否 | 外部数据集标识，仅回传，不强制生成内部 ID。 |
| `document_id` | string | 否 | 外部文档标识，仅回传，不强制生成内部 ID。 |
| `file_path` | string | 否 | 单个本地文件路径，与 `file` 至少提供一种。 |
| `file` | binary | 否 | 单文件上传，与 `file_path` 至少提供一种。 |
| `async` | boolean | 否 | 是否强制走后台任务。 |
| `page_batch_size` | integer | 否 | PDF 按页批次处理页数，默认 `100`。 |
| `enable_llm_postprocess` | boolean | 否 | 是否启用 LLM 后处理，默认开启。 |
| `return_mode` | string | 否 | 返回方式：`content`、`path`、`both`，默认 `content`。 |
| `stream` | boolean | 否 | 是否启用 SSE 流式输出。 |

### 同步响应关键字段

| 字段 | 说明 |
|---|---|
| `data.project_id` | 项目标识 |
| `data.dataset_id` | 外部数据集标识 |
| `data.document_id` | 外部文档标识 |
| `data.payload_type` | 固定为 `blocks` |
| `data.source_file_path` | 实际处理的源文档路径 |
| `data.blocks_file_path` | `return_mode=path/both` 时返回的块文件路径 |
| `data.blocks[]` | `return_mode=content/both` 时返回的块内容 |
| `data.doc_index` | 上传时创建的文档索引摘要 |
| `data.total_pages` | 总页数 |
| `data.total_blocks` | 生成块数量 |

### 异步相关接口

| 路由 | 说明 |
|---|---|
| `GET /api/data/upload_split/status?job_id=...` | 查询后台拆分任务状态 |
| `GET /api/data/upload_split/stream?job_id=...` | 订阅后台拆分流式进度 |

### 大文件异步说明

- 服务端会对较大的上传文件自动切换到后台任务模式，当前阈值为约 `20MB`
- 自动切换或显式传 `async=true` 时，根接口会立即返回任务信息，不再等待全文拆分完成
- 返回体关键字段包括：
  - `data.accepted=true`
  - `data.job_id`
  - `data.status_url`
  - `data.stream_url`
- `multipart/form-data` 直接上传文件时，不建议同时开启同路由 SSE；这类请求会按普通 JSON 响应返回任务信息，再通过 `/status` 或 `/stream` 跟踪进度

## 接口 3：块内容自动清洗

- 路由：`POST /api/data/clean`
- Content-Type：`application/json`
- 用途：清洗上传拆分后的块内容。支持路径输入、直接传块内容，也支持仅传外部 `content_id` / `dataset_id` 后从数据库读取。数据库读取默认使用表 `pipeline_payloads`。

### 请求参数

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `blocks_file_path` | string | 否 | 待清洗块文件路径，推荐。 |
| `input_file_path` | string | 否 | `blocks_file_path` 的兼容别名。 |
| `blocks` | object[] / object | 否 | 直接传入块内容；可传块数组或包含 `blocks` 的 JSON 对象。 |
| `blocks_content` | object[] / object | 否 | `blocks` 的兼容别名。 |
| `content_id` | string | 否 | 外部内容 ID；服务会从数据库读取对应块内容。 |
| `blocks_content_id` | string | 否 | `content_id` 的兼容别名。 |
| `document_id` | string | 否 | `content_id` 的兼容别名。 |
| `dataset_id` | string | 否 | 外部数据集 ID；若未传路径 / 内容 / `content_id`，则会按 `dataset_id` 读取最新块内容。 |
| `project_id` | string | 否 | 项目标识，用于补齐上下文。 |
| `block_ids` | integer[] | 否 | 兼容模式下使用；与 `dataset_id` 搭配从内部块表读取。 |
| `return_mode` | string | 否 | 返回方式：`content`、`path`、`both`，默认 `content`。 |
| `stream` | boolean | 否 | 是否开启 SSE 流式输出。可放在 JSON 体、Query `?stream=true` 或请求头 `X-Stream-Response: true`。 |

### 响应关键字段

| 字段 | 说明 |
|---|---|
| `data.project_id` | 项目标识 |
| `data.dataset_id` | 外部数据集标识 |
| `data.source_content_id` | 输入内容 ID |
| `data.payload_type` | 固定为 `cleaned_blocks` |
| `data.cleaning_rate` | 清洗命中比例 |
| `data.total_count` | 总块数 |
| `data.modified_count` | 被修改块数 |
| `data.modified_block_ids` | 被修改块 ID 列表 |
| `data.issues[]` | 清洗问题明细 |
| `data.cleaned_blocks_file_path` | `return_mode=path/both` 时返回的清洗结果文件路径 |
| `data.cleaned_blocks[]` | `return_mode=content/both` 时返回的清洗结果内容 |

### JSON 请求示例

```json
{
  "dataset_id": "ext_ds_001",
  "content_id": "ext_blocks_001",
  "return_mode": "content"
}
```

### JSON 响应示例

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "project_id": "proj_demo",
    "dataset_id": "ext_ds_001",
    "source_content_id": "ext_blocks_001",
    "payload_type": "cleaned_blocks",
    "cleaning_rate": "18.4%",
    "total_count": 38,
    "modified_count": 7,
    "modified_block_ids": [3, 8, 11],
    "issues": [],
    "cleaned_blocks_file_path": null,
    "cleaned_blocks": [
      {
        "block_id": 1,
        "page_num": 1,
        "type": "text",
        "cleaned_content": "..."
      }
    ]
  }
}
```

### SSE 流式输出

当请求体中传 `stream=true`，或 Query 参数传 `?stream=true`，或请求头传 `X-Stream-Response: true` 时，接口会改为返回 `text/event-stream`。

流式事件说明：

- `open`：SSE 连接建立成功。
- `progress`：任务已开始，本接口固定返回 `stage=started` 与“数据清洗已开始，正在处理块内容”。
- `heartbeat`：任务仍在处理中，默认约每 5 秒发送一次。
- `result`：任务完成；其中 `body` 与 `result` 字段内保存和同步模式相同的完整 JSON 返回内容。
- `error`：任务执行异常。
- `close`：流式输出结束。

### SSE 流式输出示例

```text
event: open
data: {"task":"clean_data","message":"流式连接已建立","timestamp":1777000150.101}

event: progress
data: {"task":"clean_data","stage":"started","message":"数据清洗已开始，正在处理块内容","timestamp":1777000150.102}

event: result
data: {"task":"clean_data","stage":"completed","timestamp":1777000153.208,"status_code":200,"body":{"code":200,"message":"success","data":{"project_id":"proj_demo","dataset_id":"ext_ds_001","payload_type":"cleaned_blocks","modified_count":7}},"result":{"code":200,"message":"success","data":{"project_id":"proj_demo","dataset_id":"ext_ds_001","payload_type":"cleaned_blocks","modified_count":7}}}

event: close
data: {"task":"clean_data","message":"流式输出结束","timestamp":1777000153.209}
```

## 接口 4：语义分块

- 路由：`POST /api/data/semantic_chunk`
- Content-Type：`application/json`
- 用途：把清洗后的块重组为语义块。支持路径输入、直接传块内容，也支持只传 `dataset_id` / `document_id` / `content_id` 后自动从数据库取块再分块。数据库读取默认使用表 `pipeline_payloads`。默认不再重复创建文档索引；只有显式传 `config.build_doc_index=true` 才会附带重建。

### 请求参数

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `blocks_file_path` | string | 否 | 清洗后块文件路径，推荐。 |
| `cleaned_blocks_file_path` | string | 否 | `blocks_file_path` 的兼容别名。 |
| `blocks` | object[] / object | 否 | 直接传入块内容；可传块数组或包含 `blocks` 的 JSON 对象。 |
| `cleaned_blocks` | object[] / object | 否 | `blocks` 的兼容别名。 |
| `blocks_content` | object[] / object | 否 | `blocks` 的兼容别名。 |
| `content_id` | string | 否 | 外部内容 ID；服务会从数据库读取对应块内容。 |
| `blocks_content_id` | string | 否 | `content_id` 的兼容别名。 |
| `document_id` | string | 否 | `content_id` 的兼容别名。 |
| `dataset_id` | string | 否 | 仅传此字段时，会按 `dataset_id` 自动读取块内容并直接分块；同时作为输出数据集 ID。 |
| `project_id` | string | 否 | 不传块内容时，用于从内部块表按项目回退取块。 |
| `source_block_ids` | integer[] | 否 | 仅处理指定块。 |
| `return_mode` | string | 否 | 返回方式：`content`、`path`、`both`，默认 `content`。 |
| `stream` | boolean | 否 | 是否开启 SSE 流式输出。可放在 JSON 体、Query `?stream=true` 或请求头 `X-Stream-Response: true`。 |
| `config.max_token_size` | integer | 否 | 单个语义块的 token 上限，默认 `1024`。 |
| `config.use_llm_boundary_fallback` | boolean | 否 | 分块边界回退到 LLM，默认 `true`。 |
| `config.target_protocol` | string | 否 | 仅处理某目标协议相关块。 |
| `config.target_page_window` | integer | 否 | 目标协议页邻域窗口。 |
| `config.build_doc_index` | boolean | 否 | 是否在本接口内重建索引，默认 `false`。 |

### 响应关键字段

| 字段 | 说明 |
|---|---|
| `data.project_id` | 项目标识 |
| `data.dataset_id` | 输出数据集 ID |
| `data.source_content_id` | 输入内容 ID |
| `data.payload_type` | 固定为 `semantic_chunks` |
| `data.source_blocks_file_path` | 输入块文件路径 |
| `data.chunks_file_path` | `return_mode=path/both` 时返回的语义块文件路径 |
| `data.total_chunks` | 生成语义块数 |
| `data.target_protocol` | 目标协议过滤条件 |
| `data.chunks[]` | `return_mode=content/both` 时返回的语义块内容 |
| `data.doc_index` | 仅当显式要求重建索引时返回 |

### JSON 请求示例

## 补充接口：更新文档 PageIndex 索引

- 路由：`POST /api/data/update_doc_index`
- Content-Type：`application/json` 或 `multipart/form-data`
- 用途：按文档路径或上传文件重建/补建 PageIndex 索引。`02_upload_split` 上传文档时已经会自动创建索引；当文档补传、替换或需要手工重建时，再调用这个接口。

### 请求参数

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `file` | binary | 否 | 单个文件二进制上传。适合外部服务无法直接传本机路径的场景。 |
| `files` | binary[] | 否 | 多个文件二进制上传。 |
| `document_path` | string | 否 | 单个文档路径；与 `document_paths` 至少传一个。 |
| `document_paths` | string[] | 否 | 多个文档路径；与 `document_path` 至少传一个。 |
| `project_id` | string | 否 | 兼容字段。通常不需要传；仅当同名文件跨多个项目造成歧义时，用于明确索引归属。 |

| `source_block_ids` | integer[] | 否 | 兼容字段。仅当明确只想用指定块重建时传。 |

- `file/files` 上传后，服务会先把文件保存到部署侧目录，再基于文件内容构建索引；响应中的 `project_id`、`doc_set_id`、`index_ref` 可供后续 RAG / 规则生成接口继续使用。
